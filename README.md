# conda-bashrc-optimizer

A script that automatically applies two startup-speed optimizations to `.bashrc` after installing conda (miniforge3 / Anaconda / Miniconda).

## What it does

The standard `.bashrc` block generated by `conda init` runs the following expensive operations every time you open a login shell:

1. A dynamic call to `conda shell.bash hook` (which spins up a Python interpreter and can take several seconds)
2. Automatic activation of the base environment via `conda activate base` (another few seconds)

This script applies two optimizations that dramatically reduce shell startup time (in testing, **~8 seconds → ~0.9 seconds**):

1. Runs `conda config --set auto_activate_base false` to disable automatic base activation
2. Caches the output of `conda shell.bash hook` into `~/.conda_hook_cache.sh`, and rewrites `.bashrc` to source that file instead of invoking the hook dynamically

The cache is automatically regenerated whenever the conda binary's modification time is newer than the cache file (e.g. after a conda self-update).

## Usage

Run this right after installing conda (e.g. miniforge3) on a new Linux machine:

```bash
bash optimize_conda_bashrc.sh
```

## Safety design

- **Idempotent**: does nothing if already optimized (safe to re-run)
- **Automatic backup**: backs up `~/.bashrc` to `~/.bashrc.bak.<timestamp>` before making changes
- **conda detection**: uses `command -v conda`, so it works regardless of install path (not just miniforge3)

## Verifying it worked

```bash
exec bash
```

or open a new terminal.

## Reverting

```bash
cp ~/.bashrc.bak.<timestamp> ~/.bashrc
rm -f ~/.conda_hook_cache.sh
exec bash
```

## Why this optimization is needed (background)

`conda shell.bash hook` dynamically generates a set of bash functions that let `conda activate` modify the current shell's environment variables and PS1 — something a plain subprocess can't do. Since conda itself is implemented in Python, every invocation of this command pays the cost of starting a Python interpreter. The hook's contents are essentially static as long as the conda version and install path don't change, which is exactly what makes caching effective.

Note that this is not an official conda feature — it's a workaround for a long-standing, widely reported issue in the conda community (see [conda/conda#7855](https://github.com/conda/conda/issues/7855) and similar).
